# Database Schema — Web App Lab

## Overview

This lab uses MongoDB with Mongoose. The schema design
deliberately mirrors real-world patterns — and documents
the security decisions made at the data layer.

---

## Collections

### Users Collection

Stores registered user accounts.

| Field                 | Type     | Notes                               |
| --------------------- | -------- | ----------------------------------- |
| `_id`                 | ObjectId | Auto-generated by MongoDB           |
| `username`            | String   | Unique, lowercase, trimmed          |
| `email`               | String   | Unique, lowercase, validated format |
| `password`            | String   | bcrypt hash — never plaintext       |
| `role`                | String   | Enum: `admin`, `member`             |
| `failedLoginAttempts` | Number   | Brute-force protection counter      |
| `lockUntil`           | Date     | Account lockout expiry              |
| `createdAt`           | Date     | Auto-generated by Mongoose          |
| `updatedAt`           | Date     | Auto-generated by Mongoose          |

**Security decisions:**

- Password field has `select: false` — never returned
  in queries unless explicitly requested
- Role is a fixed enum — cannot be set to arbitrary values
- Failed login tracking lives on the user document —
  persists across server restarts unlike in-memory counters

---

### Messages Collection

Stores contact form submissions.

| Field       | Type     | Notes            |
| ----------- | -------- | ---------------- |
| `_id`       | ObjectId | Auto-generated   |
| `name`      | String   | Max 100 chars    |
| `email`     | String   | Validated format |
| `message`   | String   | Max 1000 chars   |
| `createdAt` | Date     | Auto-generated   |

**Security decisions:**

- No user reference — anonymous submissions allowed
- Message content stored as-is — sanitization happens
  at the API layer before storage and display

## Security Notes

**Why passwords are hashed with bcrypt:**
Storing plaintext passwords means a database breach
exposes every user's credentials immediately. bcrypt
hashes are one-way — even with the hash, recovering
the original password requires brute-forcing.
Cost factor 12 means each hash attempt takes ~250ms —
making automated cracking computationally expensive.

**Why role is an enum:**
If role accepted any string value, mass assignment
attacks could set `role: "superadmin"` or any
arbitrary value. Restricting to a fixed set at the
schema level means invalid values are rejected before
they reach the database.

**Why select: false on password:**
A common developer mistake is querying a user and
accidentally including the password hash in an API
response. `select: false` means the field is excluded
from all queries by default — you must explicitly
opt in with `.select('+password')` when you need it,
making accidental exposure much harder.
